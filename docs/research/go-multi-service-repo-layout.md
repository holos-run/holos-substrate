# Research: Repository Layouts for Multiple Go Services (Kubernetes Controllers)

Research date: 2026-06-10. All paths were verified against the live GitHub
contents API on the projects' default branches.

## Question

The Holos Substrate repository needs to host several cooperating Go services alongside the Holos
CUE configuration: Kubernetes controllers reconciling a related set of custom
resources (the platform API), a reverse proxy that authenticates requests via
OIDC and forwards them with Kubernetes impersonation headers, the NATS webhook
receiver/subscriber/deployer pipeline ([ADR-6](../adr/archive/ADR-6.md) through
[ADR-11](../adr/archive/ADR-11.md)), and small reconcilers for Keycloak group
membership and Quay self-service resources. How do the highest quality
projects in this space lay out their repositories, and which patterns fit a
rendered-manifests (Holos) workflow?

## Projects surveyed

| Project | Repos | Go modules | Binaries | Images | API types location |
|---|---|---|---|---|---|
| Kargo | 1 | 4 (root, `api/`, generated client, tools) | 1 control-plane binary + CLI | **1** | `api/v1alpha1` (separate module) |
| Argo CD | 1 | 3 (root, `gitops-engine/`, tooling) | 1 multicall binary (argv[0]) | **1** | `pkg/apis/application/v1alpha1` (root module) |
| Crossplane | 1 | 2 (root, `apis/`) | 1 binary, subcommands | **1** | `apis/` (separate module) |
| Pinniped | 1 | 1 root + generated per-k8s-version API modules | 1 server binary (argv[0]) + CLI | **1** | `apis/` source + `generated/` |
| cert-manager | 1 | 8 (root + 5 per-binary + 2 test) | 5 binaries | 5 | `pkg/apis/` (root module) |
| external-secrets | 1 | ~45 (root, `apis/`, `runtime/`, ~41 providers) | 1 binary, subcommands + CLI | 1 | `apis/` (separate module) |
| Cluster API | 1 (+provider repos) | 3 (root, `test/`, tools) | core + 2 provider managers + CLI | several | `api/<group>/<version>` (root module) |
| Flux | ~9 (polyrepo) | 1 + `api/` per controller repo; ~30 in `fluxcd/pkg` | 1 per repo | 1 per repo | `api/` submodule per repo |

## Detailed findings

### Kargo (github.com/akuity/kargo)

The closest analog to this repository: an event-driven promotion/deployment
control plane with CRDs, controllers, webhook servers, and an API server.

- **One product container image.** A single `cmd/controlplane` binary named
  `kargo` exposes cobra subcommands — `api`, `controller`,
  `management-controller`, `garbage-collector`, `kubernetes-webhooks`,
  `external-webhooks`. Each Deployment runs the same image with a different
  subcommand. The CLI (`cmd/cli`) and a `credential-helper` are separate
  mains; the helper ships inside the same image.
- **API types in a separate low-dependency module**: `api/go.mod` holds
  `api/v1alpha1` CRD types plus protobuf service definitions, so external
  consumers import types without Kargo's full dependency tree.
- **Implementation lives in `pkg/`** (51 packages: `pkg/controller`,
  `pkg/server`, `pkg/webhook`, `pkg/promotion`, ...). No `internal/` on
  current main; earlier versions used `internal/` and later consolidated.
- **CRDs are generated into the deployment artifact**: `make codegen-controller`
  runs controller-gen and emits CRDs directly into the Helm chart at
  `charts/kargo/resources/crds`. The generated manifests are part of the
  shipped configuration, not a side output — the same shape as a Holos
  component consuming generated CRDs.
- Tooling: `hack/tools` module + `hack/tools.mk` install pinned tools into
  `hack/bin`; `buf` for proto; `Tiltfile` and k3d/kind scripts under `hack/`.

### Argo CD (github.com/argoproj/argo-cd)

- **One image, one multicall binary.** `cmd/main.go` dispatches on
  `filepath.Base(os.Args[0])`: `argocd`, `argocd-server`,
  `argocd-application-controller`, `argocd-repo-server`,
  `argocd-applicationset-controller`, etc. Deployments symlink the binary to
  the component name.
- Components are top-level packages in one root module (`server/`,
  `controller/`, `reposerver/`, `applicationset/`); `pkg/` is reserved for
  the public surface (`pkg/apis`, `pkg/apiclient`).
- Manifests are kustomize under `manifests/` with pre-rendered install YAMLs
  regenerated by `hack/update-manifests.sh`.

### Crossplane (github.com/crossplane/crossplane)

- **Two modules**: root + `apis/` with deliberately minimal dependencies
  (`k8s.io/api`, `apimachinery`, `controller-runtime` only), wired locally by
  a `replace` directive.
- **One binary** (`cmd/crossplane`) with `core`, `rbac`, and `render`
  subcommands; one image; the core and RBAC managers are two Deployments of
  the same image with different args.
- **No `pkg/` at all** — everything non-API is `internal/` (17 packages,
  including `internal/controller/<area>` and `internal/webhook/`). The public
  Go surface is exactly the `apis/` module.

### Pinniped (github.com/vmware-tanzu/pinniped)

Directly relevant prior art for the OIDC/impersonation proxy.

- **One server image; one combined server binary** (`cmd/pinniped-server`)
  dispatched by argv[0] and symlinked to `pinniped-concierge`,
  `pinniped-supervisor`, and `local-user-authenticator` in the image. The CLI
  (`cmd/pinniped`) ships separately as a kubectl credential plugin.
- The **impersonation proxy** lives in `internal/concierge/impersonator/`
  (proxy + roundtripper), with its lifecycle managed by a controller in
  `internal/controller/impersonatorconfig/` — i.e. the proxy is itself
  *modeled and managed through the Kubernetes API*, matching ADR-2's
  KRM-first principle.
- API types: `apis/` is the source of truth; `generated/<k8s-minor>/`
  contains per-Kubernetes-version API+client modules with their own `go.mod`
  so integrators take only the types.
- jetstack/kube-oidc-proxy (archived 2024) is the other prior art for an
  OIDC→impersonation-headers reverse proxy: classic small single-module
  layout (`cmd/`, `pkg/proxy/`, `deploy/`, `hack/`).

### cert-manager (github.com/cert-manager/cert-manager)

The counter-example on binaries: **five binaries, five images**, and a Go
module per binary (`cmd/controller/go.mod`, `cmd/webhook/go.mod`, ...) so
each image carries a pruned dependency tree and its own LICENSES file. The
cost is eight modules to maintain and a vendored Makefile-module system
(`klone.yaml`) to keep the build coherent. API types remain in the root
module under `pkg/apis/`. The `controller` binary is a multi-controller
manager with one package per control loop under `pkg/controller/`.

### external-secrets (github.com/external-secrets/external-secrets)

- One controller binary with subcommands (`controller`, `cert-controller`,
  `webhook`) — three Deployments, one image — plus an `esoctl` CLI.
- A 2025 refactor split the repo into ~45 modules: `apis/`, `runtime/`, and
  one module per provider, to keep each provider SDK's heavy dependency tree
  out of consumers. This is what module-per-component maintenance looks like
  at scale (~50 `replace` directives in the root go.mod).

### Cluster API (github.com/kubernetes-sigs/cluster-api)

- Many cooperating managers in **one repo, one root module**: the core
  manager at `/main.go`, kubeadm bootstrap and control-plane provider
  managers under `bootstrap/kubeadm/` and `controlplane/kubeadm/`, the CLI
  under `cmd/clusterctl/`.
- In v1.11 all API types were consolidated to a top-level
  `api/<group>/<version>` tree (e.g. `api/core/v1beta2`,
  `api/bootstrap/kubeadm/v1beta1`) guarded by `.import-restrictions` to keep
  API packages dependency-light — but still **inside the root module**; the
  extra modules (`test/`, `hack/tools/`) exist only to isolate heavy
  test/tool dependencies.

### Flux (fluxcd) — the polyrepo pattern

- One repo per controller (source-controller, kustomize-controller,
  helm-controller, notification-controller, image-*-controller), each a
  standalone kubebuilder project with its own image and an `api/` submodule
  so consumers import only types. `fluxcd/flux2` composes them: the CLI, the
  distribution manifests, and a pinned bill-of-materials of controller
  versions. Shared code lives in `fluxcd/pkg` (~30 independently tagged
  modules).
- The split exists because Flux is explicitly a *toolkit* — third parties
  run subsets and build on individual controllers.
- Observable costs: sequenced multi-repo releases (pkg → each controller →
  flux2), a version compatibility matrix in every release note, dependency
  bump churn (`runtime v0.108.0` — triple-digit minors from constant bumps),
  no atomic cross-controller refactors, and N copies of CI/Makefile/release
  scaffolding.

## Conventions and official guidance

- **Kubebuilder multi-group layout**: `api/<group>/<version>/*_types.go`,
  reconcilers in `internal/controller/<group>/`, webhooks in
  `internal/webhook/<group>/<version>/`, one `cmd/main.go`, one
  controller-runtime Manager running all controllers, kustomize under
  `config/`. Kubebuilder's good-practices guidance is one controller per
  Kind (single responsibility); it does not call for separate managers or
  Deployments per controller.
- **golang-standards/project-layout is not a standard.** Russ Cox:
  "the *vast* majority of packages in the Go ecosystem do *not* put the
  importable packages in a `pkg` subdirectory... It is unfortunate that this
  is being put forth as 'golang-standards' when it really is not"
  ([project-layout#117](https://github.com/golang-standards/project-layout/issues/117)).
  Official guidance is [go.dev/doc/modules/layout](https://go.dev/doc/modules/layout).
- **Single module unless versions must diverge.** Official Go guidance
  ([managing module source](https://go.dev/doc/modules/managing-source)):
  hosting a single module at the repository root "will help keep maintenance
  simpler"; multi-module repos exist for *independent versioning* and impose
  prefixed version tags (`api/v1.2.3`) per module. `go.work` is a local
  development convenience, not a publishing mechanism.

## Patterns that emerge

1. **Monorepo with one image and one multi-role binary is the dominant
   pattern for cohesive control planes.** Kargo, Argo CD, Crossplane,
   Pinniped, and external-secrets all converge on it, despite arriving from
   different histories. Subcommands (Kargo, Crossplane, external-secrets) and
   argv[0] multicall (Argo CD, Pinniped) are interchangeable mechanics; the
   point is one build, one image, N Deployments with different args.
2. **API types are isolated for cheap import** — either as a separate
   low-dependency module (Kargo, Crossplane, external-secrets, Flux) or as a
   dependency-restricted package tree in the root module (Cluster API,
   cert-manager, Argo CD). Every project does the isolation; only some pay
   the multi-module tag/release overhead, and several (Kargo, Crossplane,
   external-secrets) extracted the module *later*, once external consumers
   existed.
3. **Generated CRDs land in the deployment artifact** (Kargo → Helm chart;
   Argo CD → `manifests/`; kubebuilder → `config/crd`). Codegen output is
   wired into whatever renders the manifests — for this repository, that is a
   Holos component.
4. **Auxiliary integrations are controllers, not CLIs.** Pinniped manages
   even its own impersonation proxy with a controller. The equivalent here:
   Keycloak group membership and Quay repositories become CRDs with small
   reconcilers in the same manager, rather than imperative tools.
5. **Polyrepo (Flux) and image-per-binary (cert-manager) solve problems we
   do not have** — independent third-party consumption of components, and
   dependency pruning for binaries injected into user workloads — and both
   carry heavy, permanent coordination costs.

## Candidate layouts for this repository

- **Option A — Single module, single binary, subcommand-per-service**
  (Kargo/Crossplane/Pinniped shape): kubebuilder multi-group conventions,
  one `cmd/`, `api/<group>/<version>`, `internal/`, one image; Holos CUE in
  `holos/`.
- **Option B — Single module, one binary and image per service**
  (cert-manager shape without per-binary modules): `cmd/<service>/` each
  built to its own image.
- **Option C — Multi-module monorepo**: root module plus `api/` module now
  (and possibly per-service modules), `go.work` for development.
- **Option D — Polyrepo** (Flux shape): keep/extend `holos-controller` as a
  separate repo, one repo per service, this repository holds only CUE.

The ranking of these options against the project's goals at the time and
the decision are recorded in [ADR-12](../adr/ADR-12.md).
